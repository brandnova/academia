# Governance

This document explains how Academia is owned, how decisions get made, and how contributors are recognized. It exists so expectations are clear from day one, for everyone's benefit.

## Project relationship to Brand Nova

Academia is a standalone product, conceived, designed, and built from its first line of code by Ijeoma M. Jahsway, founder of Brand Nova. It is not a community-founded or community-owned project, it is an existing, complete product under the Brand Nova umbrella and its founder has chosen to open to outside contributions.

## Roles

**Founder & Lead Maintainer (Ijeoma M. Jahsway):**
- Retains final decision-making authority over the project's architecture, roadmap, feature direction, branding, and documentation.
- Owns the GitHub organization and repository administration (branch protection, releases, deployment configuration, access management).
- Reviews and merges all pull requests, or designates a co-maintainer to do so as the project grows.
- Can update this governance document as the project and team evolve.

**Contributors:**
- Anyone who opens an issue, submits a pull request, improves documentation, or otherwise helps the project, once that contribution is merged or accepted.
- Contributing to Academia does not grant ownership, equity, or any legal right to the Academia or Brand Nova name, brand, or trademark.
- Contributors retain no special claim over the direction of the project beyond the normal process of proposing and discussing changes through issues and pull requests.

This structure is intentional, not exclusionary. Academia was built end to end by one person before any outside contributor was involved, and the project needs one clear point of final decision-making to stay coherent, especially while most contributors are new to collaborative development. As the contributor base grows and proves itself over time, this document can and likely will evolve, for example, by adding trusted contributors as co-maintainers with merge rights.

## How decisions get made

- **Small changes** (bug fixes, small features, documentation fixes, minor UI polish): open an issue or pick up an existing one, discuss briefly if needed, submit a pull request. The maintainer reviews and merges.
- **Larger changes** (new features not already on the roadmap, architectural changes, anything that touches the documented API contract or data model): must be discussed in an issue first, before any code is written. The maintainer will confirm whether and how it fits before work begins. This matches the project's existing API-first, documentation-first development approach.
- **Disagreements** about approach are resolved by discussion first. If no agreement is reached, the maintainer makes the final call. This isn't about being right, it's about keeping the project moving without decisions stalling indefinitely.

## Brand and trademark

The "Academia" name, the "Brand Nova" name, and any associated logos, wordmarks, or visual identity are **not** covered by the project's code license (see LICENSE). They remain the property of the founder. Forking the code is permitted under the license terms, but forks may not use the Academia or Brand Nova name or branding to represent themselves as this project or as affiliated with Brand Nova without permission.

This is a normal and common split in open source projects: the code is open, the brand identity is not. It protects the project's identity while still letting the code itself be freely used, learned from, and built upon.

## Code license

The Academia codebase is released under the MIT License (see LICENSE in the repository root). In short: anyone can use, copy, modify, and distribute the code, including commercially, provided the original copyright and license notice is retained. The license applies to the code only, not to the project's name or branding, as described above.

## Contributor recognition

Every contributor whose work is merged is recognized in the following ways:
- Listed automatically on GitHub's own contributor graph for the repository.
- Credited in the project's README and, once it exists, a dedicated CONTRIBUTORS section or page.
- Recognized as a contributor to a Brand Nova product, which they're welcome to reference in their own portfolios, resumes, or LinkedIn profiles ("Contributed to Academia, a Brand Nova product").

Recognition is based on merged, meaningful contributions. Opening an issue or a draft PR that isn't ultimately merged is genuinely appreciated, but formal recognition is tied to work that ships.

## Changes to this document

This document reflects where the project is today: early in its collaboration journey, with one maintainer and a small group of new contributors. As the project and team grow, this document will be revisited and updated, with contributors notified of any changes that affect them.
